# AI Coding 文档工作流

## 目标

让 Codex、CodeBuddy、其他模型和人工工程师在同一套事实下工作。文档不是任务结束后的补记，而是需求、决策、实现和合并过程的一部分。

## 各阶段更新时机

| 阶段 | 必须检查或更新 | 完成标志 |
| --- | --- | --- |
| 需求进入 | PRD、ROADMAP、已有 Decisions | 范围、验收标准、任务 ID 和依赖明确 |
| 方案设计 | TECH、共享 Schema、DECISIONS | 数据契约和模块边界先于消费代码确定；重大取舍已有 ADR |
| 开工认领 | GitHub Issue；不可用时创建 `handoffs/APR-*.md` | 负责人、分支、允许修改模块、风险和基线 SHA 已记录 |
| 编码进行中 | 模块 README、API/Schema 注释、handoff | 行为或命令变化当场同步，不依赖模型记忆留到最后 |
| 提交 PR 前 | PRD/TECH/部署文档、handoff、测试证据 | 文档与代码一致；记录已运行命令、未完成项、head SHA |
| 合并后 | STATUS | 仅集成负责人把已合并能力写入 STATUS，并更新日期 |
| 发布前后 | DEPLOYMENT、README、发布检查结果 | 启动命令、环境变量、回滚/降级和线上验证均可复现 |
| 方向废弃 | docs/README、archive/ | 从主线索引移除，迁入归档并说明替代方案 |

## 变更类型与文档门禁

- 改用户流程、文案原则或范围：必须更新 PRD。
- 改路由、数据流、评分逻辑、隐私边界或服务拓扑：必须更新 TECH；难以逆转时先写 ADR。
- 改 JSON 字段或 API：Schema、TypeScript/Python 类型、fixture 和技术文档必须同 PR 更新。
- 改脚本、命令、环境变量或部署：必须更新对应 README 或 DEPLOYMENT。
- 只做内部重构且外部行为、契约、命令均不变：可以不改产品文档，但 PR 需明确“文档无变化”的理由。

## AI 智能体开工检查

1. 按 `AGENTS.md` 的事实来源顺序阅读相关文档。
2. 用 `rg` 检查旧路径和重复说明，确认本任务应修改的唯一权威文档。
3. 若文档冲突，停止编码并在 DECISIONS 中登记，不自行选择更方便的版本。
4. 开工时声明预计更新的文档；交付时列出实际更新项和无需更新项。
5. 不得把计划中、分支中或 mock 的能力写成 `main` 已交付能力。

## 文档责任边界

- 任务负责人：维护本任务的 Issue/PR 或 handoff，并同步相关产品、架构和操作文档。
- 契约负责人：保证 Schema、类型、fixture 和接口说明一致。
- 集成负责人：合并后更新 STATUS，处理跨任务冲突。
- 发布负责人：维护 DEPLOYMENT 和发布验证结果。

## 归档标准

当文档描述已删除产品、已取消技术路线，或其事实已被新的权威文档完整替代时，应迁入 `docs/archive/`。归档文件保留原名以便 Git 历史追踪，并在 `docs/README.md` 标明其非权威性质。
